# GPT 검토 의견 - HWPX 문서 표현 엔진 고도화 우선순위

작성일: 2026-06-30

## 검토 대상

사용자가 한글 2024에서 직접 연 다음 baseline 문서를 시각적으로 검토했다.

```text
release/test-documents/army-claw-hwpx-capability-baseline.hwpx
```

## 현재 판정

현재 Worker는 한컴 2024 호환 템플릿을 기반으로 HWPX 파일을 생성하고, 제목과 본문을 넣으며, 구조 검증을 통과하는 단계까지 도달했다. 사용자가 제공한 화면에서도 문서가 한글 2024에서 열리고 한글 텍스트, 제목, 목차, 5개 본문 절이 표시되는 것을 확인했다.

따라서 다음은 1차 통과로 본다.

- HWPX 파일 생성
- 한컴 2024 네이티브 열기
- 한글 텍스트 표시
- DocumentPlan 내용 전달
- 제목과 본문 섹션 출력

다만 현재 결과물은 실무용 보고서가 아니라 텍스트를 한 페이지에 순서대로 배치한 초안 수준이다. 파일 호환성과 내용 전달은 성공했지만 문서 표현 엔진은 아직 완성되지 않았다.

## 화면에서 확인한 문제

### 1. 표지, 목차, 본문이 분리되지 않음

표지 제목, 부제, 작성 정보, 목차, 본문이 한 페이지에 연속 배치되어 있다. 다음 페이지 구조가 필요하다.

```text
1페이지: 표지
2페이지: 목차
3페이지 이후: 본문
```

표지와 목차 뒤에는 실제 page break를 사용해야 한다.

### 2. 제목 계층이 시각적으로 약함

문서 제목을 제외하면 `1. 검증 목적`, `2. 기능 구성` 등의 장 제목이 본문과 거의 같은 크기와 간격으로 보인다. 최소한 다음 스타일이 구분되어야 한다.

```text
cover_title
cover_subtitle
cover_metadata
toc_title
toc_item
heading_1
heading_2
heading_3
body
```

### 3. 표가 실제 표 객체가 아님

기능별 검증 결과가 공백과 텍스트 나열로 표시된다. 열 정렬, 테두리, 줄바꿈과 셀 폭을 보장하려면 실제 `hp:tbl` 객체가 필요하다.

다음 기능을 우선 구현한다.

- 표 제목
- 머리글 행
- 데이터 행
- 임의 행·열 수
- 셀 텍스트
- 기본 테두리
- 셀 내부 여백
- 머리글과 본문 스타일 구분
- 문서 폭을 넘지 않는 열 너비
- 긴 텍스트 줄바꿈

고급 셀 병합과 중첩 표는 이후로 미룬다.

### 4. 본문 가독성이 부족함

본문 글자 크기와 줄 간격이 작고, 문단 및 장 사이 간격이 거의 없다. 한 페이지에 지나치게 많은 내용이 압축되어 있다.

권장 목표:

- 본문 10~11pt 수준
- 줄 간격 150~170% 수준
- 제목 앞뒤 간격 확보
- 문단 아래 간격 확보
- 목록의 들여쓰기와 내어쓰기 구분

실제 HWPX 단위는 템플릿의 기존 글자 모양과 문단 모양을 우선 재사용하고, 존재하지 않는 스타일 ID를 임의로 참조하지 않는다.

### 5. 메타정보가 일반 본문처럼 출력됨

부제, 부서, 작성자와 날짜를 표지의 별도 메타정보 영역으로 배치해야 한다. 본문 시작 전에 일반 문장처럼 연속 출력하지 않는다.

### 6. 목차가 정돈되지 않음

현재 목차는 단순 번호 목록이다. 자동 목차 필드가 안정적이지 않다면 정적 목차를 유지하되 다음을 구현한다.

- 목차 제목 스타일
- 항목별 들여쓰기
- 항목 간격
- 가능하면 점선 리더와 페이지 번호

### 7. callout이 일반 문단과 구분되지 않음

`key_result` 같은 강조 블록은 실제 시각 요소로 보여야 한다. 1행 1열 표를 재사용하는 방법도 허용한다.

필수 요소:

- 제목
- 테두리
- 내부 여백
- 본문과 구분되는 스타일
- 가능하면 옅은 음영

### 8. 꼬리말이 실제 footer가 아님

현재 `꼬리말:` 문장이 본문 마지막에 출력된다. 이 텍스트는 실제 footer 영역으로 이동해야 하며 `꼬리말:`이라는 라벨은 최종 문서에 나타나지 않아야 한다.

### 9. 페이지 번호가 없음

본문 페이지 하단에 실제 페이지 번호 객체를 넣는 방안을 구현한다. 구조를 충분히 검증하지 못한 경우 지원했다고 보고하지 말고 제한사항으로 남긴다.

## 구현 우선순위에 대한 GPT 의견

이번 단계에서는 backend 실행 큐 연결보다 문서 표현 엔진의 품질을 먼저 완성하는 것이 타당하다.

권장 순서:

```text
1. 실제 hp:tbl 표 객체
2. 문서 스타일 체계
3. 표지·목차·본문 page break
4. callout 시각화
5. 실제 목록 문단과 들여쓰기
6. 머리말·꼬리말·페이지 번호
7. 동일 DocumentPlan으로 v2 생성
8. 한글 2024 사용자 시각 검증
9. backend와 Node Worker Adapter 연결
10. 실행 큐 및 패키징 연결
```

이 순서의 이유는 문서 품질이 낮은 상태에서 실행 큐부터 연결하면 낮은 품질의 문서를 더 편리하게 반복 생성하는 기능만 먼저 완성되기 때문이다. Worker의 출력 품질을 먼저 검증하고 그 다음 Army Claw 실행 흐름에 연결해야 한다.

## 구현 방법에 대한 원칙

- HWPX XML 구조를 추측으로 만들지 않는다.
- 한글 2024가 실제 생성한 비민감 HWPX 구조를 기준으로 구현한다.
- 설치된 한컴 원본 템플릿은 GitHub에 커밋하지 않는다.
- 실제 군 문서, 기관 문서, 로고와 개인정보를 fixture로 사용하지 않는다.
- 템플릿의 `Contents/header.xml`에 정의된 글자 모양, 문단 모양, borderFill을 우선 재사용한다.
- 새 style ID가 필요하면 header와 section의 참조 무결성을 함께 검증한다.
- LLM은 DocumentPlan만 만들고 Worker가 결정론적으로 HWPX를 렌더링한다.
- baseline 파일은 삭제하지 않고 v2와 비교할 기준으로 보존한다.

## 다음 v2 완료 기준

동일한 baseline DocumentPlan으로 다음 파일을 생성한다.

```text
release/test-documents/army-claw-hwpx-capability-v2.hwpx
```

v2는 최소 다음을 만족해야 한다.

- 표지 페이지가 독립됨
- 목차 페이지가 독립됨
- 본문이 별도 페이지에서 시작됨
- 제목 1~3과 본문 스타일이 구분됨
- 실제 `hp:tbl` 표 객체가 존재함
- 표 머리글과 본문이 구분됨
- callout이 시각적으로 구분됨
- 문단 및 목록 들여쓰기가 정돈됨
- 구조 검증 오류가 없음
- 한글 2024에서 정상적으로 열림

머리말, 꼬리말과 페이지 번호는 가능한 범위에서 구현하되, 한글 2024 구조 검증을 통과하지 못하면 완료로 간주하지 않는다.

## 한셀·한쇼 확장 고려

향후 한셀과 한쇼도 같은 실행 구조를 사용해야 한다. 공통 프로세스 실행, JSON 입출력, timeout, 로그와 오류 변환은 포맷 독립적인 `HancomWorkerAdapter`로 설계하고 HWPX 전용 렌더링 규칙만 별도 모듈로 둔다.

```text
HancomWorkerAdapter
├─ HWPX renderer
├─ HCell/XLSX worker
└─ HShow/PPTX/SHOW worker
```

이번 HWPX 표현 엔진 구현이 한셀과 한쇼 Tool을 막는 과도한 HWPX 전용 결합을 만들지 않도록 한다.

## Codex 작업 시 필수 확인

Codex는 다음 구현을 시작하기 전에 이 문서를 읽고, 자신의 구현 계획에 반영한 항목과 반영하지 않은 항목을 먼저 요약해야 한다.

작업 종료 보고서에는 다음을 포함한다.

- 이 GPT 의견에서 반영한 항목
- 기술적 이유로 반영하지 못한 항목
- 실제 구현된 HWPX 객체와 스타일
- baseline과 v2의 구조 차이
- 한글 2024 사용자 확인 필요 항목
